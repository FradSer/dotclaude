# Superpowers Memory Layer — Design Index

## Context

The superpowers plugin shipped, today, a "docs index" convention (`lib/docs-index.sh` + project-root `docs/README.md`): a single controlled-vocabulary table (`kind ∈ {design, plan, retro}`) that four skills — brainstorming, writing-plans, executing-plans, retrospective — consult before they start work and update after they commit. It answers "what artifact folders exist, and are they still authoritative?" It does not answer "what did we *learn*?" Separately, `retrospective`'s existing Pre-Check B already taps a second, unrelated memory concept: this assistant's own private, cross-project, harness-injected `~/.claude/.../memory/MEMORY.md`, read as an advisory calibration signal scoped to retrospective alone, never fetched on demand, and invisible to teammates or other sessions.

Neither system captures a distilled, reusable fact — "task files must copy design scenario titles verbatim, or the plan-reflection check false-positives," "`repo_root()` silently targets the parent repo when `CLAUDE_PROJECT_DIR` is unset" — in a form the *other four* skills, and especially `systematic-debugging` (which has zero `docs/` touchpoints of any kind today), can deterministically read before they start. The user's request: make `docs/README.md` double as a genuine memory layer for all five superpowers skills — read before every operation, updated after any of them — designed by reference to Claude Code's own memory system (one fact per file, a lightweight index of one-line pointers, Why/How-to-apply bodies), and, non-negotiably, never allowed to bloat.

This design extends the shipped docs-index convention with a fourth `kind=memory` rather than building a parallel mechanism, and gates every write on a threshold each skill already tracks internally — no new counters are invented anywhere in this design.

## Discovery Results

- **Docs-index is shipped and working** (commits `c848b9e`..`66c73cf`, today). `lib/docs-index.sh` implements `list`/`show`/`upsert`/`set-status`/`rebuild` against one `docs/README.md` table; `kind` validation is enforced at every write site with a hard `design|plan|retro` allowlist.
- **Anti-bloat is already solved once, at the index level, not the content level.** `docs/plans/2026-07-04-docs-index-design/best-practices.md` §Anti-Bloat Rules sets a 60-line ceiling on `docs/README.md` with a two-stage collapse rule (merge ≥3 same-status/topic rows; if still over budget, drop `expired` rows entirely, keeping only their tombstone in the retro that expired them). This budget is shared across all kinds and this design extends it to `memory` rather than carving out a second ceiling.
- **Four of five skills already have a docs-index touchpoint** (consult-before + upsert-after): brainstorming, writing-plans, executing-plans, retrospective. **`systematic-debugging` has zero `docs/` touchpoints today** — confirmed by grep across its `SKILL.md`. It does already carry a hard-coded escalation trigger: "Architecture Questioning After 3+ Failed Fixes" (`skills/systematic-debugging/SKILL.md:241,309,333`).
- **Each of the other four skills already has its own rework/escalation threshold**, independently established before this design, none of which needs to be invented:
  - `brainstorming` — "REWORK 2+ rounds: consider pivoting back to Phase 1" (`SKILL.md:135`).
  - `writing-plans` — Phase 4 sub-agent reflection is the sole plan-quality gate; a FAIL forces fix-and-rerun before the Phase 5 commit (`SKILL.md:204`).
  - `executing-plans` — the intra-plan-learning "variety gap" signal: *"If all checklist items PASS for a batch but the batch required 2+ rework rounds"* (`references/intra-plan-learning.md:54`). A related-but-distinct signal exists in `references/batch-execution-playbook.md:165` ("Maximum 2 evaluation-rework rounds before escalation") — that is a hard execution-abort cap (batch gives up and logs a HARD BLOCKER), not a "this eventually passed but the friction is worth remembering" signal, and is deliberately NOT reused as the memory-write gate (see Glossary).
  - `retrospective` — the existing ADD (2+ distinct plans) / MODIFY (2+ false positives) proposal thresholds (`SKILL.md:119-124`).
- **Retrospective's Pre-Check B is real and already shipped** (`SKILL.md:31-41`): it scans the harness-injected `MEMORY.md` already present in context at session start, treating it as a secondary/advisory signal subordinate to `docs/retros/evolution-log.jsonl`. This is the only place any superpowers skill reads any memory concept today, and it is private, global, and harness-injected — not project-local, not git-tracked, not queryable on demand.
- **A stray, untracked `superpowers/docs/README.md` currently sits on disk** (one row: `docs/writing-skills/ | retro | reference`). It predates the 2026-07-04 relocation (commit `d6b8d49`) that moved project-docs artifacts from the plugin's own `superpowers/docs/` to the consuming project's root `docs/`. It contradicts the shipped architecture and is not git-tracked; deleted as routine cleanup in this design's Phase 3 commit.

## Glossary

Canonical labels for this design. Two real cross-sub-agent divergences were found and reconciled during integration; rejected variants are recorded below.

| Concept | Canonical label | Rejected variants | Why canonical |
|---|---|---|---|
| Per-file classification field (frontmatter) | `category` | `type` (assistant's own memory schema uses this) | `type` collides with the index row's reserved `kind`/existing vocabulary; `category` is unambiguous at both levels |
| Category enum values | `convention \| pitfall \| decision \| preference` | assistant's own `user \| feedback \| project \| reference` | Superpowers facts are skill-generated engineering lessons, not assistant/user-relationship facts; `reference` specifically is rejected as a category value because `status=reference` already exists at the row level for evergreen docs — reusing the word at the file level would create two "reference" vocabularies in one feature |
| Memory file path pattern | `docs/memory/<category>_<slug>.md` | flat `docs/memory/<slug>.md` (used loosely in early drafts) | Mirrors the assistant's own memory file naming exactly (`feedback_*.md`, `project_*.md`) — directly satisfies "design by reference to Claude Code's memory system"; also makes `ls docs/memory/pitfall_*` a grep-free category filter |
| Memory row's allowed status values | `active \| expired:<reason>` only, **enforced by the script** | "any of the 6 shared statuses, convention-only (not code-enforced)" | The plugin's own established norm is code-enforced controlled vocabulary, never a documentation-only convention (see shipped req #8: "MUST enforce the controlled status vocabulary... exits 2, writes nothing"); a memory fact is never mid-pipeline (`wip`), never "shipped" (`implemented:<sha>` doesn't apply), never a positive replacement pointer (`superseded-by:<path>` — consolidation drops the absorbed row outright instead), and never evergreen-sticky (`reference` — memory is designed to expire, unlike `docs/writing-skills/`) |
| executing-plans' memory-write trigger | `intra-plan-learning.md:54` "variety gap" (2+ rework rounds, batch eventually PASSes) | `batch-execution-playbook.md:165` "max 2 rounds before escalation" | The playbook's cap is a hard execution-abort signal (batch gives up, HARD BLOCKER logged) — a different event from "this succeeded, but the friction it took is a reusable lesson." The variety-gap signal is the precise semantic match for a memory-worthy insight; the escalation cap is retained as run-time flow control only, not repurposed as a second write-gate (avoids double-triggering on what retrospective's own cross-plan mining already covers) |
| Retrospective's memory-write scope | `ADD` and `MODIFY` proposals only | including `REMOVE`/`PROMOTE` | `REMOVE` is a retraction ("this checklist item wasn't needed") — a negative signal, not a distilled fact worth keeping. `PROMOTE` is a checklist-internal capability graduation, not a project-level insight. Including either would let the checklist's own "only ever grows" failure mode leak into the memory store |
| Frontmatter provenance field | `source` (repo-relative path, `commit:<sha>`, or omitted) | duplicating `status` inside the file's own frontmatter | `status` lives only on the index row (single source of truth, matching how `_index.md`/retro reports never carry a duplicate status field either); `source` is a new, distinct, always-file-level field with no row-level counterpart to collide with |

## Requirements

### Functional (MUST)

15. **MUST** extend the `kind` controlled vocabulary in `lib/docs-index.sh` to `design|plan|retro|memory`, validated at every existing enforcement site (`validate_kind`, the `cmd_list --kind` filter check) — no kind value bypasses validation.
16. **MUST** store each distilled fact/decision/convention/pitfall in its own file at `docs/memory/<category>_<slug>.md` — one discrete unit per file. An aggregate or multi-fact file is explicitly disallowed; this is the core anti-bloat mechanism (see Rationale).
17. **MUST** give every memory file frontmatter with `name`, `category` (enum: `convention|pitfall|decision|preference`), `summary` (≤72 chars), `source` (repo-relative path, `commit:<sha>`, or omitted), `created`, `updated`. `category` MUST NOT reuse the reserved words `kind` or `type`.
18. **MUST** record only a single index-row pointer per memory file in the *existing* `docs/README.md` table (`path`, `kind=memory`, `status`, `summary`, `updated`) — no memory file's body content is ever inlined into `docs/README.md`.
19. **MUST** enforce, at the script level, that a `kind=memory` row's `status` is restricted to `active | expired:<reason>` — `upsert`/`set-status` attempting `wip`, `implemented:<sha>`, `superseded-by:<path>`, or `reference` on a `memory` row exits 2 and writes nothing.
20. **MUST** add a read-before step to the entry point of all five skills — brainstorming, writing-plans, executing-plans, systematic-debugging, retrospective — running `lib/docs-index.sh list --kind memory --status active`, then reading the topically-relevant files, before the skill produces its first artifact or code change.
21. **MUST** add a conditional write-after step to all five skills, gated on each skill's own **existing** escalation threshold (brainstorming: 2+ evaluator REWORK rounds; writing-plans: a Phase 4 reflection FAIL requiring rework; executing-plans: the intra-plan-learning "variety gap," 2+ rework rounds; systematic-debugging: its existing 3+ failed-fixes trigger, or an explicit cross-cutting gotcha; retrospective: its existing ADD/MODIFY thresholds). **MUST NOT** invent a new threshold where an existing one already serves the purpose.
22. **MUST** keep systematic-debugging's write step a single conditional step folded into its existing completion turn, not a new phase or commit obligation — its "fix + regression test, never a planning artifact" contract is unchanged, and this remains its only `docs/` touchpoint.
23. **MUST NOT** alter retrospective's existing Pre-Check B behavior beyond one addition: Phase 3 MAY promote a recalled global-memory prior into a `docs/memory/<category>_<slug>.md` file, citing the originating hook, when the prior proves project-specific and durable. The private recall itself stays advisory-only, harness-injected, and read nowhere else in the plugin.
24. **MUST NOT** permit secrets, credentials, tokens, or PII in any `docs/memory/*.md` file — these files are git-tracked and team-shared; the writing skill is responsible for this check at write time (no new script-level content filter — see Rationale).

### Functional (SHOULD)

25. **SHOULD** consolidate 2+ memory files addressing the same concept into one file, via retrospective (reusing its existing 2+-instance MODIFY threshold shape), before the shared ceiling forces collapse.
26. **SHOULD** flip a memory row's status to `expired:<reason>` when a later change invalidates the fact — including as the first step of a consolidation merge (the absorbed file is tombstoned, then dropped) — rather than silently deleting the file.
27. **SHOULD** physically move an `expired:<reason>` memory file to `docs/memory/archive/` when its row is dropped from the index under the existing second-line collapse defense, so the fact stays discoverable rather than becoming an untracked orphan. `rebuild`'s memory scan (a non-recursive `docs/memory/*.md` glob) naturally excludes `archive/` without any extra code.
28. **SHOULD** keep each memory row's `summary` field to the existing single-physical-line, ≤72-char, diff-friendly convention — no format exception relative to `design`/`plan`/`retro` rows.

### Non-functional

29. **MUST NOT** raise or duplicate the existing 60-line ceiling on `docs/README.md` — `memory` rows share the same budget and the same two-stage collapse rule as every other kind.
30. **MUST** remain parseable with `grep`/`awk` alone — a memory row is a `docs/README.md` table row like any other; a memory file is a markdown file with YAML frontmatter like every other artifact this plugin produces. No new tool or schema is introduced.

## Rationale

**Extend docs-index rather than build a parallel memory system.** A second index would mean two files to consult before every skill runs, two things to keep in sync, and two collapse rules to maintain — exactly the drift risk `lib/docs-index.sh` exists to eliminate. `kind=memory` is a fourth row-type in a system that already has the validation, the collapse rule, and the touchpoint pattern proven out; the marginal cost is one enum value plus one small kind-aware status restriction, not a new subsystem.

**Reuse each skill's existing threshold rather than invent new ones.** Every skill already has its own calibrated escalation point, each tuned against this plugin's own real history (brainstorming's REWORK-pivot check, writing-plans' Phase 4 FAIL gate, executing-plans' variety-gap signal, systematic-debugging's 3-strikes rule, retrospective's ADD/MODIFY thresholds). A sixth, memory-specific "write when X" rule bolted on top would either fire more cheaply than the skill's own quality bar (writing near-noise into `docs/memory/`) or duplicate a check the skill already performs — inventing a new threshold here is exactly the kind of unjustified addition this plugin's own anti-add-bias norm exists to catch (see this week's docs-index retro, which self-rejected two ADD proposals for 1-plan evidence).

**One fact per file is the actual anti-bloat mechanism — not a stylistic preference.** A single growing `docs/memory.md` has no natural collapse point: every new fact is an append inside one file, invisible to the 60-line *row* ceiling (which counts index rows, not lines inside a target file). Splitting one-fact-per-file makes the *index*, not the file, the sole growth surface: each fact costs exactly one `docs/README.md` row, so the shipped ceiling and its collapse rule apply to memory exactly as they already apply to designs and plans, with zero new logic. This is the direct, mechanical answer to "don't let this README bloat" — bloat is structurally forced through the same accounted-for row budget every other artifact already obeys.

**Complementary to, not duplicative of, the private global memory.** `~/.claude/.../memory/MEMORY.md` is harness-injected once per session, cross-project, and read only by retrospective's Pre-Check B as an advisory prior — no mechanism exists for the other four skills to query it on demand, and what little of it surfaces is never scoped to their specific task. `docs/memory/` closes that gap deterministically: project-local, git-tracked, team-shared, and explicitly `Read`/`Bash`-accessible by all five skills whenever they choose to consult it. Retrospective is the one skill positioned to bridge the two, since it already reads global memory and already promotes findings into durable state; the other four skills have no private-memory read path to begin with, so `docs/memory/` is their only memory surface, by construction — there is nothing to duplicate.

**Anticipated question: "Doesn't this duplicate the memory system retrospective already reads?"** No — the two differ on every axis that matters. Pre-Check B's memory is private (this assistant instance only), global (spans every project on the machine), harness-injected (arrives automatically, never fetched on demand), and advisory-only (subordinate to `evolution-log.jsonl` on disagreement). `docs/memory/` is project-local (scoped to this repo), git-tracked (any collaborator or future session reads it), explicitly accessed (a deliberate read-before step), and authoritative within its scope (a recorded fact is as durable as an `implemented:<sha>` design row). Pre-Check B ships unchanged; this design adds one promotion bridge, not a merge of the two systems.

**No script-level secret/PII filter.** A generic secret-shape regex would false-positive constantly on legitimate technical content (SHA hashes, example config values, sample tokens in a pitfall writeup). Validation-at-mutation-time is already this plugin's pattern (`validate_kind`/`validate_status` run before any write) — but content-policy review is qualitatively different from format validation, and is left as the writing skill's authorial responsibility, same as it already is for every other git-committed doc this plugin produces.
