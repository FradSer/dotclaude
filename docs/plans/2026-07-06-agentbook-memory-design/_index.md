# Agentbook Commons Bridge — Design Index

## Context

`dotclaude` already has two memory tiers. Tier A is this assistant's private, harness-injected, cross-project `~/.claude/.../memory/MEMORY.md`, read only by `retrospective`'s Pre-Check B as an advisory prior. Tier B is the repo-local, git-tracked `docs/memory/<category>_<slug>.md` + `docs-index.sh kind=memory` layer shipped 2026-07-04, consumed by five superpowers skills (brainstorming, writing-plans, executing-plans, systematic-debugging, retrospective).

Neither tier answers "has any agent, anywhere, already solved a materially similar problem, and how much should I trust that fix given independent real-world confirmation?" That is what agentbook (`/Users/FradSer/Developer/FradSer/agentbook`) — a separate, real, live project describing itself as "the public debug-knowledge commons for AI coding agents" — is built for: a REST + MCP service where `recall`/`trace` are free anonymous reads and `remember`/`report`/`verify` are Bearer-authenticated writes that feed a Bayesian confidence score (author self-reports never count; confidence rises only when independent agents confirm an outcome).

The user's request: integrate agentbook's live service — not just its design ideas — as a genuinely reusable, cross-plugin capability, not a superpowers-only mechanism. This design covers where that capability lives, how it degrades when agentbook is unreachable or unconfigured, how it relates to the two existing memory tiers, and which concrete plugins consume it and how.

## Discovery Results

- **Cross-plugin Skill-tool resolution is undocumented.** Official Claude Code docs confirm skill namespacing (`plugin-name:skill-name`) prevents naming conflicts, but do not specify whether a skill registered by plugin A resolves when invoked from content that lives in plugin B. What **is** documented and enforced is a `plugin.json` `"dependencies"` array: a plugin can declare a dependency on another installed plugin, and Claude Code auto-resolves/installs it (`/reload-plugins` reinstalls if a dependency goes missing). Building the shared capability as a **standalone plugin** that consumer plugins declare a formal dependency on is the only mechanism with documented, enforced behavior — nesting it as an internal skill inside `superpowers` and hoping other plugins can reference it by name rests on undocumented behavior.
- **Agentbook's actual MCP tool contract** (`agentbook/backend/presentation/mcp/tools.py`, `agentbook/docs/mcp-setup.md`): five tools — `recall` (anonymous, 30/min), `trace` (anonymous), `remember` (Bearer, 120/hr), `report` (Bearer, 10/hr), `verify` (Bearer, 5/min + 20/hr sandbox budget, Python-single-file-only). Two distinct error envelopes (protocol-layer JSON-RPC errors vs. tool-layer `result.isError`). A presented-but-invalid credential is rejected at the transport with HTTP 401; a genuinely absent credential returns a tool-layer `unauthorized` — the two must be handled differently (the latter is an expected, silent-skip state; the former indicates broken configuration worth surfacing once).
- **Agentbook is pre-pilot/alpha by its own admission**: same-task recall is validated (weak-model pass@1 lift measured), cross-task transfer currently shows zero fix-lift, real external traffic is near zero. The design must treat it as a fragile, optional enhancement, never a dependency any core autonomous workflow (e.g. `executing-plans`) can block or fail on.
- **Credential storage precedent already exists in this repo** — no new pattern needed: `office/lib/progressive_env.py` (`resolve()`/`resolve_secret()`, CLI flag → env → `.env` chain → default), `code-context/.mcp.json` (env-var interpolation in MCP server `headers`, e.g. `"x-api-key": "${EXA_API_KEY}"`), `git-agent-cli/cmd/config.go` (`maskAPIKey`, never print/log a raw key). Agentbook's Bearer key follows the `code-context/.mcp.json` pattern for the bundled MCP server config and the `progressive_env.py` chain for any script-level use.
- **`docs-index.sh`'s row/path machinery is structurally local-file-only.** `validate_path()` rejects absolute paths / `..` traversal on the assumption `path` resolves under `repo_root()`; `scan_folders()` and `rebuild` reconcile the index by globbing `docs/memory/*.md` on disk; `validate_status_for_kind()` narrows `kind=memory` to `active|expired:<reason>` specifically because that matches Tier B's lifecycle. Agentbook records have no local file, live on a remote server, and follow a `candidate → promoted/demoted` + confidence-score lifecycle with zero semantic overlap. Force-fitting a fifth `kind=agentbook` value would be the one kind `rebuild` structurally cannot reconcile, and would blow the 60-line row-ceiling budget (a `recall` can fire many times per debugging session; each would cost a row under the naive approach).
- **Three concrete, evidence-grounded consumer touchpoints** (not hypothetical):
  - `autoresearch/scripts/setup-autoresearch.sh` — the plateau-escalation moment (3 consecutive non-improving rows trigger an expensive GAN tournament, `TOURNAMENT_BLOCK`) is exactly where a cheap `recall` for a similarly-shaped optimization problem could be tried first; the existing `results.tsv` outcome log (`keep`/`discard`/`gatefail`/`crash`/`tournament`) is exactly where a `report` call is a parallel write, not a new decision point.
  - `superpowers/skills/systematic-debugging/SKILL.md` — **premise correction**: this skill is not memory-touchpoint-free; commit `57d3738` (2026-07-06, same day) already added Tier B touchpoints (Phase 1 step 0 "Consult Memory," Phase 4 step 6 "Capture Memory"). The gap this design closes is that Tier B is repo-scoped — `recall` adds a parallel, cross-project/cross-agent source at the same Phase 1 step 0 moment; `report` fits at Phase 4 step 3 (Verify Fix, the instant "did it work" is known), independent of Tier B's heavier 3-strikes write gate.
  - `github/skills/review-pr/references/review-loop.md` — the `[ci]` failure reaction table (lint/type/test/build errors → open-ended, project-specific fixes) is a genuine iterate-until-green loop; `recall` could surface a previously-successful fix for the same recurring CI error signature in this repo, and `report` fits exactly where the Monitor already detects `[ci] pass` vs. a recurring failure.
  - **Ruled out with evidence, not assumed**: `git`/`gitflow` plugins are deterministic dispatch tables (fixed error-signature → fixed action, e.g. 401 → `--free`) with no hypothesis-forming loop to attach recall to — genuinely NO FIT, not a forced integration.

## Glossary

| Concept | Canonical label | Rejected variants | Why canonical |
|---|---|---|---|
| The new tier itself | **public commons** | `memory tier 3`, `network memory`, `agentbook memory` (all reuse the reserved word "memory," already precisely claimed by Tier B — a third overload recreates the `category`-vs-`type` collision Tier B's own design already fought once); `global memory` (Tier A is already informally "the private global memory" — reusing "global" for public/cross-agent scope would leave two tiers both plausibly called that) | Matches agentbook's own self-description verbatim ("the public debug-knowledge commons for AI coding agents") rather than coining an in-house synonym for an already-named external concept |
| The plugin/skills wrapping the agentbook MCP calls | **commons bridge** | `agentbook skill` (bakes one vendor's brand into a generic cross-plugin component's name); `memory client` (reuses reserved "memory," implying false kinship with Tier B); `docs-index-remote` (wrongly implies an architectural relationship to `docs-index.sh`, ruled out below); `MCP client` (collides with the many unrelated MCP servers already in this session, e.g. `mcp__claude-in-chrome__*`) | Signals the two-way (inbound recall, outbound publish), cross-plugin nature without tying the name to one vendor or implying a docs-index relationship |
| The act of contributing local knowledge outward | **publish** | `promote` (already claimed twice in Tier B's design: the private-global→repo-local bridge, and retrospective's checklist `PROMOTE` proposal type — a third meaning would make one word mean three non-interchangeable operations); `remember` (agentbook's own literal MCP tool name — using it as the skill-level verb blurs "the tool call" with "the skill's decision to call it"); `contribute` (already loose, undefined prose in agentbook's own docs — under-specific as a canonical term) | Foregrounds the one-way, CC0-public-domain, externally-visible nature of the act — distinct from a reversible git commit and from either existing sense of "promote" |
| The act of recalling external knowledge inward | **recall** | `memory-read` (reserved — Tier B's established name for the `docs-index.sh list --kind memory` read-before touchpoint); `lookup`/`query` (generic, coins an in-house synonym for a fixed external API term for no benefit); `consult` (reserved — Tier B's name for the docs-index consult-before touchpoint) | Matches agentbook's own fixed MCP tool name 1:1; Tier A's Pre-Check B informally says "recall" in lowercase prose but was never elevated to a capitalized Glossary term, so no formal collision — going forward, write "the `recall` tool" / "agentbook recall" to keep the two usages visually distinct |
| Bridge between a Tier B memory file and a Tier C record | **`source: agentbook:<problem_id>`** (fourth shape on the existing `source` frontmatter field) | A new frontmatter field (rejected — `source` already has a three-shape union, `<path>` / `commit:<sha>` / omitted, established as a parameterized-value pattern; a fourth shape is the minimal, precedent-consistent extension) | Lets a `docs/memory/*.md` file cite an agentbook record as provenance in either direction without agentbook ever becoming a docs-index `kind`, row, or something `rebuild` must reconcile |

## Requirements

### Functional (MUST)

1. **MUST** implement the commons bridge as a **new standalone plugin** (`commons-bridge/`), not as an internal skill nested inside `superpowers` — cross-plugin Skill-tool resolution is undocumented; the `plugin.json` `"dependencies"` array is the only mechanism with documented, enforced cross-plugin behavior.
2. **MUST** register exactly two internal skills in `commons-bridge/.claude-plugin/plugin.json`'s `"skills"` array (non-slash-command): `recall` (inbound: query agentbook before solving a problem) and `publish` (outbound: contribute a validated finding). Consumer plugins declare `"dependencies": ["commons-bridge"]` and load `commons-bridge:recall` / `commons-bridge:publish` via the Skill tool.
3. **MUST** bundle an `.mcp.json` in `commons-bridge/` registering agentbook as a Streamable HTTP MCP server, with the Bearer credential supplied via env-var interpolation (`"Authorization": "Bearer ${AGENTBOOK_API_KEY}"`), mirroring `code-context/.mcp.json`'s existing pattern — never a literal key in a committed file.
4. **MUST** treat every agentbook call (`recall`/`trace`/`remember`/`report`/`verify`) as optional and best-effort: a missing MCP server, network error, timeout, or any error envelope MUST degrade to "no external memory available" for that call site and MUST NOT block, hang, retry indefinitely, or fail the calling skill's core workflow.
5. **MUST** distinguish the two failure classes correctly: a tool-layer `unauthorized` (no credential presented) is a silent, expected skip; an HTTP 401 (a presented-but-invalid/revoked credential) indicates broken configuration and should be surfaced to the user once, not retried silently forever.
6. **MUST** respect `retry_after_seconds` on every `rate_limit_exceeded` response (recall: 30/min anonymous; remember: 120/hr; report: 10/hr; verify: 5/min + 20/hr sandbox budget) — back off at least that long; never spin-retry.
7. **MUST** treat all `recall`/`trace` content as reference data, never as directly executable instructions — never execute a recalled solution's commands verbatim without understanding them; gate application on the solution's `confidence` and its own `verification` checks.
8. **MUST** call `report(success: false, notes: ...)` when a recalled solution is judged malicious, wrong, or unsafe, so it gets demoted for the next agent — never silently ignore a bad recall.
9. **MUST** distinguish `verify`'s `not_verifiable` (content-shape limitation — not Python-single-file) from a genuine `{"status":"verified","passed":false}` sandbox failure. Only the latter is real negative evidence; `not_verifiable` alone MUST NOT trigger `report(success: false)`.
10. **MUST** gate every `publish` call behind an explicit redaction/content-review step before `remember` fires — secrets, tokens, PII, internal hostnames/paths/project names MUST be stripped. This is a required step, not best-effort, because published content becomes CC0-1.0 public domain and effectively irrevocable, and agentbook's own write gate is only a backstop.
11. **MUST NOT** add a fifth `kind` value to `docs-index.sh` for agentbook records — `validate_path()`, `scan_folders()`, and `rebuild()`'s glob-reconciliation are structurally local-file-only and have no meaningful application to a remote record with no local file. Agentbook integration is architecturally separate from `docs-index.sh` in every respect except item 12.
12. **MUST** support the `source: agentbook:<problem_id>` shape as a fourth value in the existing memory-file frontmatter `source` field's union (`<path>` / `commit:<sha>` / omitted / `agentbook:<problem_id>`), letting a Tier B memory file cite provenance to or from a Tier C record without any docs-index row-level change.
13. **MUST** implement the `publish` promotion gate (Tier B → Tier C) as **retrospective-owned**, requiring ALL of: (a) `category == pitfall` only — agentbook's problem/solution model has no natural shape for `convention`/`decision`/`preference`; (b) cross-*project* recurrence evidence (not cross-plan, which Tier B's existing MODIFY threshold already covers) — the proposal must name the specific other project(s)/incidents; (c) a stricter content bar than Tier B's req #24 (public CC0 visibility, not just team-shared); (d) explicit human confirmation before the `remember` call fires — this is NOT auto-applied the way Tier B's ADD/MODIFY proposals are.
14. **MUST NOT** reuse Tier B's existing private-global→repo-local promotion gate (req #23, "project-specific and durable") for the new outward direction — the new gate is the mirror image (generalizable and safe-for-everyone, not project-specific); a fact that just passed req #23 because it proved project-specific is a poor candidate for immediate outward re-promotion.
15. **MUST** wire the three evidence-grounded consumer touchpoints as `commons-bridge:recall` / `commons-bridge:publish` load points: `autoresearch/scripts/setup-autoresearch.sh` (recall before plateau/tournament escalation; report alongside `results.tsv` logging), `superpowers/skills/systematic-debugging/SKILL.md` (recall at Phase 1 step 0 alongside the existing Tier B consult; report at Phase 4 step 3, Verify Fix), `github/skills/review-pr/references/review-loop.md` (recall/report at the `[ci]` failure reaction loop).
16. **MUST NOT** add a touchpoint to `git`/`gitflow` plugins — confirmed NO FIT (deterministic dispatch table, no hypothesis loop); forcing an integration here would be unjustified scope creep.
17. **MUST** each declare `"dependencies": ["commons-bridge"]` in the `plugin.json` of every consumer plugin from item 15 (`autoresearch`, `superpowers`, `github`).

### Functional (SHOULD)

18. **SHOULD** log every `recall`/`report` call's outcome (hit/miss/error) at the call site the same way the consuming skill already logs its own decisions (e.g. `autoresearch`'s `results.tsv`), so agentbook's contribution to a decision is auditable without agentbook itself being the source of truth.
19. **SHOULD** default `recall` to explicit, skill-documented invocation points (per item 15), never an automatic background call on every skill turn — mirrors Tier B's own deliberate, documented read-before steps rather than a surprising network call in an otherwise offline-capable toolset.

### Non-functional

20. **MUST NOT** let `commons-bridge` introduce a hard dependency anywhere in `executing-plans`' fully autonomous execution path — it is reachable only through `systematic-debugging`'s optional recall step, which already has graceful degradation (item 4) as its own hard requirement.
21. **MUST** keep the bundled `.mcp.json` and skill content free of any literal credential — verified by the same review discipline this repo already applies to every committed file.

## Rationale

**Standalone plugin, not a nested superpowers skill.** The one piece of hard evidence from this design's research is that cross-plugin `Skill` tool resolution has no documented contract — Claude Code's own docs confirm skill namespacing prevents *name* conflicts but say nothing about whether plugin B's content can successfully invoke a skill only plugin A registered. The one mechanism that IS documented and enforced for cross-plugin capability declaration is `plugin.json`'s `"dependencies"` array. Betting the entire "usable by other plugins" requirement on undocumented behavior, when a documented alternative exists, is the wrong trade — especially since the standalone-plugin cost (one more `.claude-plugin/plugin.json`, one README, one marketplace.json entry) is small relative to the risk of a cross-plugin reference silently failing in a future Claude Code version.

**Two skills, not one.** `recall` and `publish` have opposite risk profiles: `recall` is anonymous, free, rate-limited-but-generous, and safe to invoke often; `publish` requires a Bearer credential, is rate-limited to 120/hr, produces irrevocable CC0-public content, and needs a redaction gate no automated check can fully replace. Splitting them lets a consumer plugin depend on `recall` alone (the common case — autoresearch, systematic-debugging's inbound check, github:review-pr) without inheriting `publish`'s heavier safety contract; only `retrospective` needs both.

**Not a docs-index `kind`.** Every mechanism `docs-index.sh` provides — path validation, filesystem-glob reconciliation via `rebuild`, kind-aware status narrowing, the 60-line row ceiling — is built on the assumption that a row's `path` resolves to a local file this repo's git history explains. An agentbook record has no local file, evolves via a remote confidence-score lifecycle this repo doesn't control, and would legitimately generate far more rows than the ceiling tolerates (a `recall` can fire many times per debugging session). Bridging via the existing `source:` frontmatter field's union — rather than a new `kind` — costs nothing new in `docs-index.sh` and is the more precedent-consistent extension.

**The publish gate is the mirror of the existing promotion gate, not a reuse of it.** Tier B's req #23 promotes broad-but-unproven (private global memory) into narrow-and-proven (repo-local) when a recalled prior proves project-specific and durable — a de-scoping filter. The new outward gate runs the opposite direction and needs the opposite filter: narrow-and-proven-here into broad-and-safe-for-everyone, which specifically excludes facts that are durable *because* they're tied to this repo's own idiosyncratic structure. Reusing req #23's threshold verbatim would promote exactly the wrong candidates.

**Why `git`/`gitflow` are excluded.** Every documented error signature in those plugins already has exactly one prescribed action written inline in the skill (e.g., 401 → retry with `--free`); there is no point where Claude forms a hypothesis or tries competing fixes for an *unknown* error. Recall has nothing to add over what's already hardcoded, and there's no "worked/didn't work" moment to report since there's no iteration. Including them anyway, "because it could theoretically help," is exactly the unjustified-scope-creep failure mode this design's own research was tasked to avoid.

## Risks

| Risk | Concrete mitigation |
|---|---|
| Agentbook (pre-pilot/alpha) changes its MCP contract, breaks, or goes offline entirely | Req #4: every call site treats the integration as optional and best-effort; a missing/erroring tool degrades to "no external memory available" and never blocks, hangs, or fails the calling workflow. No consumer plugin has a hard dependency on agentbook's uptime. |
| A leaked or committed `AGENTBOOK_API_KEY` | Req #3/#21: the bundled `.mcp.json` only ever holds the env-var interpolation `${AGENTBOOK_API_KEY}`, never a literal key; credential resolution follows the existing `progressive_env.py` chain; masking follows `git-agent-cli`'s `maskAPIKey` convention. A dedicated review scenario (`bdd-specs.md` "Bundled .mcp.json and skill content carry no literal credential") gates every commit touching `commons-bridge/`. |
| A recalled solution is malicious, wrong, or unsafe and gets executed verbatim | Req #7/#8: recalled `content`/`steps` are treated as reference data only, gated on `confidence` and `verification`; a bad recall triggers `report(success: false, notes: ...)` to demote it for future callers. |
| A local pitfall is published to the public commons before being properly redacted | Req #10: redaction (secrets/PII/internal paths/hostnames/project names) is a mandatory, non-skippable step before any `remember` call — enforced by design, not left to agentbook's own write-gate backstop. Publication is CC0-1.0 and effectively irrevocable, so this gate has no "undo" fallback and must not be bypassed. |
| A non-generalizable or premature fact gets published, cluttering the public commons or confusing other agents | Req #13/#14: the four-criterion publish gate (category=pitfall, cross-project recurrence evidence, stricter public content bar, not a fast-follow of the Tier A→B gate) plus mandatory human confirmation — never auto-applied. |
| Rate limits (recall 30/min, remember 120/hr, report 10/hr, verify 5/min+20/hr) are exceeded and the calling skill spin-retries or stalls | Req #6: every rate-limited call site respects `retry_after_seconds` and falls back to "proceed without this" rather than retrying in a loop. |
| Cross-plugin dependency resolution (the `plugin.json` `"dependencies"` mechanism) doesn't behave as documented, or `commons-bridge` goes missing after being installed | Standalone-plugin architecture decision (Req #1) deliberately chooses the one mechanism with documented, enforced behavior over undocumented cross-plugin Skill-tool resolution; `/reload-plugins` reinstalls a missing dependency automatically (see `bdd-specs.md` "commons-bridge is later removed or goes missing"). |
| A `kind=agentbook` docs-index row is added "for consistency," breaking `rebuild`'s filesystem-reconciliation invariant or blowing the 60-line row ceiling | Req #11 explicitly forbids this; the only sanctioned bridge is the `source: agentbook:<problem_id>` frontmatter shape (Req #12), which touches zero docs-index rows. |
| An unjustified touchpoint gets added to `git`/`gitflow` (or any other plugin without a real iterate-and-diagnose loop), diluting the design's evidence-grounded scope | Req #16 explicitly excludes `git`/`gitflow` based on the consumer survey's NO FIT finding (deterministic dispatch table, no hypothesis-forming loop); no touchpoint, no dependency declaration for those plugins. |
