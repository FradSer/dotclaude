# Designing Loops — Best Practices

## Pitfalls specific to loop-selection guidance itself

**Over-recommendation (escalating past what the task needs) is over-engineering.** Recommending `/schedule` for genuinely one-off work, or `/goal` for a single-file mechanical edit, imposes cron/turn-cap/condition-verification overhead the task never needed. This mirrors the bail-out philosophy already used throughout this plugin: `README.md`'s "When NOT to use superpowers" table (`skill | bail-out trigger | override`) is the precedent pattern `designing-loops` should imitate — an equivalent `loop type | over-recommendation trigger | correct classification` table, rather than defaulting to "escalate when in doubt." `skills/brainstorming/SKILL.md`'s own Bail-Out Check explicitly warns that defaulting to "proceed when in doubt" biases an agent single-direction toward over-engineering; the same warning applies here. This is REQ-008's justification in `_index.md`.

**Under-recommendation (staying at plain turn-based for genuinely unbounded/multi-turn work) under-delivers.** The mirror failure: telling Claude to "just judge for itself when it's done" for a task with 5+ sequential steps and a real acceptance criterion abandons the entire reason `/goal` exists — explicit completion criteria plus a turn cap, so Claude neither stops early nor runs away. `skills/references/goal-wrapper.md`'s own existence is the argument against this pitfall: it was built precisely because turn-based self-judgment doesn't reliably converge on genuinely multi-step work.

**Time-based conflation risk.** Anyone extending or reviewing `designing-loops` must not conflate the native `/loop` command (still active, session-level, interval-driven) with this plugin's own deleted v2.x continuation runtime (formerly `lib/loop.sh`, removed in v3.0.0 per `README.md`'s "Removed in v3.0.0" note, replaced by native `/goal`). These are different mechanisms with different owners (platform vs. plugin) and different histories; the skill body must name them unambiguously. This is REQ-006's regression case, directly exercised by `bdd-specs.md`'s "Claude never conflates native `/loop` with the deleted v2.x runtime" scenario.

**Known content gap, flagged rather than silently patched:** `skills/references/goal-wrapper.md` documents only *condition* phrasing (its Rule 2 verifiable/unverifiable table) — it has no turn-cap phrasing guidance, despite the blog explicitly calling out "explicit turn caps ('stop after 5 tries')" as part of managing a goal-based loop. `designing-loops` surfaces this gap and carries the guidance itself as original content (REQ-023), with an inline note that the durable fix belongs in `goal-wrapper.md` (the shared reference all 5 command skills already consult) — not duplicated per-skill, and not silently fixed in this design's scope (see `_index.md` Risks for why the edit itself stays deferred).

## CRITICAL-marker placement (apply when writing `SKILL.md`)

Any load-bearing classification rule in `designing-loops`' `SKILL.md` body must carry an explicit `CRITICAL:` marker in the L2 body itself — never only in an L3 reference file. Per the recorded lesson (`feedback_skill_level_enforcement.md`, corroborated by this repo's own precedent of upgrading soft-worded rules in gitflow/github/swiftui to explicit `CRITICAL:` blocks, and most recently by the v3.5.0 postmortem on `verification-before-completion`/`receiving-code-review`'s own load-bearing "MUST load" instructions living only in L3): rules stated only as `## Note:`, `## Pre-operation Checks`, or soft prose get treated as optional and skipped — both sub-agents and the main agent follow the first matching L2 path and rarely descend into L3 detail unless explicitly directed there.

Concretely for `designing-loops`, three rules are load-bearing enough to require an L2 `CRITICAL:` block (REQ-012): the four-way exhaustive classification itself, the trivial-work bail-out (REQ-008), and the citation-not-duplication constraint (REQ-009/REQ-010). A references file with a more detailed decision tree is a legitimate L3 companion, but is not a substitute for the L2 marker — an agent moving fast will not descend into it unless L2 already told it the rule exists.

## Forward pointer: how skill-content quality itself gets tested

No `skill-creator` plugin exists inside this repo (`dotclaude`) — it is available only as an externally-registered skill outside this marketplace, so there is currently no in-repo eval mechanism for skill-content *behavioral* quality, as distinct from `plugin-optimizer`'s structural/token-budget validation (which checks form, not whether the guidance actually generalizes). The external skill-creator's own methodology is the applicable pattern for future adoption: benchmark a skill's behavioral performance with variance analysis across repeated runs (a single run can pass by chance), testing whether the skill's guidance generalizes to novel inputs rather than whether a validating agent can reconstruct the expected answer from context it was leaked.

This is noted as a forward pointer only — not built as part of this design. When this repo eventually needs a testable "did Claude classify these N loop-selection prompts correctly" harness, that external pattern (repeated-run variance benchmarking over held-out classification prompts) is the applicable approach — analogous to how `superpowers-evaluator`'s binary-checklist mechanism already tests design/code artifacts but has no equivalent today for a skill's own behavioral accuracy.

## Security / performance

No new hooks, no new environment variables or state flags, no new external network calls, no new synchronous LLM invocation on any Stop/PostToolUse/UserPromptSubmit hook critical path — `designing-loops` is documentation-only content loaded on demand like every other internal skill. The only "performance" surface is the existing SessionStart bootstrap cost, which this design does not add to (§3.1 of `architecture.md` deliberately keeps `designing-loops` out of the `using-superpowers` routing table `hooks/session-start.sh` scrapes, so it carries zero session-start token cost — it loads only when its own description triggers a match).
